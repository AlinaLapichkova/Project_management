## ФАЗА 3

## Описание проекта

По итогам Фазы 1 был выбран проект "**Финансовый помощник с прогнозированием (BudgetGuard)**"


Акторы системы: Пользователь, Банковский API / Банк, Push-уведомления, Email-сервис, Платёжные системы

## Use Cases

### UC-1. Регистрация пользователя
*Акторы*: Пользователь

*Цель*: Создать новый аккаунт в приложении

*Предусловие*: Пользователь не авторизован

*Основной сценарий*: 
- Пользователь открывает приложение
- Нажимает «Регистрация»
- Вводит email и пароль (или использует социальные сети)
- Подтверждает email по ссылке
- Система создаёт аккаунт и авторизует пользователя
- Система перенаправляет пользователя на главный экран

*Негативный сценарий*: Неверный формат email или слабый пароль — система показывает ошибку и предлагает исправить

*Альтернативный сценарий*: Регистрация через Google/Apple ID

*Постусловие*: Пользователь зарегистрирован, авторизован и видит главный экран

### UC-2. Авторизация в приложении
*Акторы*: Пользователь

*Цель*: Войти в существующий аккаунт

*Предусловие*: У пользователя есть зарегистрированный аккаунт

*Основной сценарий*: 
- Пользователь открывает приложение
- Вводит email и пароль
- Нажимает «Войти»
- Система проверяет данные и открывает главный экран

*Негативный сценарий*: Неверные данные — система предлагает «Восстановить пароль»

*Альтернативный сценарий*: Вход через Google/Apple ID или биометрию (Face ID/Touch ID)

*Постусловие*: Пользователь авторизован, отображается его бюджет и история

### UC-3. Добавление банковской карты
*Акторы*: Пользователь, Банковский API

*Цель*: Подключить карту для автоматического импорта транзакций

*Предусловие*: Пользователь авторизован

*Основной сценарий*: 
- Пользователь заходит в «Настройки → Банки»
- Выбирает банк
- Проходит авторизацию в банке (OAuth)
- Система получает доступ к транзакциям
- Транзакции импортируются автоматически

*Негативный сценарий*: Ошибка подключения (банк недоступен) — система предлагает повторить или ручной ввод

*Альтернативный сценарий*: Подключение нескольких карт

*Постусловие*: Карта подключена, транзакции синхронизируются регулярно

### UC-4. Ручной ввод транзакции
*Акторы*: Пользователь

*Цель*: Добавить расход или доход вручную

*Предусловие*: Пользователь авторизован

*Основной сценарий*: 
- Пользователь нажимает «+ Добавить транзакцию»
- Выбирает тип (расход/доход)
- Вводит сумму, категорию, дату, комментарий.
- Нажимает «Сохранить»
- Система добавляет запись в историю

*Негативный сценарий*: Некорректные данные (отрицательная сумма) — система показывает ошибку

*Альтернативный сценарий*: Быстрый ввод голосом или шаблоном

*Постусловие*: Транзакция добавлена, бюджет и прогноз обновлены

### UC-5. Категоризация транзакций
*Акторы*: Пользователь, Система

*Цель*: Присвоить категорию транзакциям (автоматически или вручную)

*Предусловие*: Есть транзакции в истории

*Основной сценарий*: 
- Система предлагает категорию на основе описания/правила
- Пользователь подтверждает или изменяет категорию
- Система сохраняет правило для будущих похожих транзакций

*Негативный сценарий*: Пользователь игнорирует — транзакция остаётся «Без категории»

*Альтернативный сценарий*: Массовое редактирование категорий

*Постусловие*: Транзакции категоризированы, статистика и прогноз точнее

### UC-6. Установка месячного бюджета
*Акторы*: Пользователь

*Цель*: Задать лимиты расходов по категориям

*Предусловие*: Пользователь авторизован

*Основной сценарий*: 
- Пользователь заходит в раздел «Бюджет»
- Выбирает месяц
- Устанавливает лимиты по категориям
- Нажимает «Сохранить»

*Негативный сценарий*: Превышение общего лимита: пользователь устанавливает сумму лимитов больше ожидаемого дохода — система показывает предупреждение «Сумма лимитов превышает доход на X руб.» и предлагает скорректировать

*Альтернативный сценарий*: Копирование бюджета с предыдущего месяца: пользователь нажимает «Копировать из прошлого месяца», система подгружает лимиты и позволяет их скорректировать

*Постусловие*: Бюджет установлен, приложение отслеживает прогресс и превышения

### UC-7. Просмотр прогноза бюджета
*Акторы*: Пользователь

*Цель*: Получить прогноз расходов на месяц

*Предусловие*: Есть история транзакций (минимум 1 месяц)

*Основной сценарий*: 
- Пользователь открывает главный экран или раздел «Прогноз»
- Система показывает прогноз на основе привычек, регулярных платежей и лимитов
- Пользователь может просмотреть детали (графики, категории)

*Негативный сценарий*: Недостаточно данных для прогноза: у пользователя менее 1 месяца истории — система показывает сообщение «Для точного прогноза нужна история хотя бы за 1 месяц» и предлагает ручной ввод ожидаемых трат

*Альтернативный сценарий*: Ручная корректировка прогноза: пользователь добавляет разовую трату/доход (например, премия или крупная покупка), система мгновенно пересчитывает прогноз

*Постусловие*: Пользователь видит ожидаемые остатки и предупреждения о рисках

### UC-8. Получение уведомлений о превышении
*Акторы*: Пользователь, Push-уведомления, Email-сервис

*Цель*: Предупредить о приближении или превышении лимита

*Предусловие*: Бюджет установлен, уведомления включены

*Основной сценарий*: 
- Система фиксирует превышение лимита (например, 90% или 100%)
- Отправляет push/email-уведомление с деталями
- Пользователь открывает уведомление и переходит в приложение

*Негативный сценарий*: Уведомления отключены: пользователь ранее выключил уведомления — превышение происходит без предупреждения

*Альтернативный сценарий*: Уведомление с предложением действий: в уведомлении кнопки «Посмотреть детали», «Скорректировать лимит», «Игнорировать на сегодня»

*Постусловие*: Пользователь осведомлён о превышении, может скорректировать расходы

### UC-9. Генерация отчёта за период
*Акторы*: Пользователь, Email-сервис

*Цель*: Получить аналитику расходов за выбранный период

*Предусловие*: Есть транзакции за период

*Основной сценарий*: 
- Пользователь заходит в «Отчёты»
- Выбирает период
- Система генерирует графики, таблицы и рекомендации
- Пользователь экспортирует в PDF или отправляет по email

*Негативный сценарий*: Нет данных за выбранный период: пользователь выбирает будущий месяц или период без трат

*Альтернативный сценарий*: Сравнение с другим периодом: пользователь выбирает «Сравнить с прошлым месяцем/годом» — система показывает динамику

*Постусловие*: Отчёт сформирован и доступен пользователю

### UC-10. Оформление премиум-подписки
*Акторы*: Пользователь, Платёжные системы

*Цель*: Приобрести премиум-подписку для доступа к расширенным функциям

*Предусловие*: Пользователь авторизован, в приложении есть премиум-функции

*Основной сценарий*: 
- Пользователь заходит в «Настройки → Премиум»
- Просматривает преимущества премиум-подписки
- Выбирает тариф (месячный/годовой)
- Нажимает «Оформить подписку»
- Система перенаправляет в платёжный интерфейс (Платёжные системы)
- Пользователь вводит данные карты или использует сохранённый способ оплаты
- Платёжные системы подтверждают платёж
- Система активирует премиум-статус и показывает подтверждение

*Негативный сценарий*: платёж отклонён (недостаточно средств, ошибка карты) — система показывает ошибку и предлагает повторить или выбрать другой способ.

*Альтернативный сценарий*: пробный период (7–14 дней бесплатно)

*Постусловие*: Премиум-подписка активирована, пользователь получает доступ к расширенным функциям, платёж зафиксирован

## Состав команды 

При оценке проекта в Use Case Points было получено 102,6 UСP, что равняется **2052,6 чел/ч**, при условии расчета 20 чел/ч = 1 UСP (оптимистичный сценарий для средней команды)

С учетом сложности проекта, оптимальной будет команда из 5 человек:
- Frontend-разработчик (Middle) - 140 000 руб/мес
- Backend-разработчик (Middle) - 140 000 руб/мес
- Дизайнер (Junior) - 80 000 руб/мес
- Тестировщик (Middle) - 100 000 руб/мес
- Проектный менеджер (Middle) - 90 000 руб/мес

## Оценка трудоемкости и сроков разработки 

С учетом того, что рабочее время одного человека в месяц = 8 ч/д * 20 рабочих дней = 160 чел/ч, команда из 5 человек обладает ресурсом ~ 800 чел/ч. С поправкой на итеративность разработки, встречи, коммуникации и перерывы (~ 30% времени) - оценка сроков разработки ~ 3,7 месяца. 

Итоговый срок реализации проекта: **4 месяца**
